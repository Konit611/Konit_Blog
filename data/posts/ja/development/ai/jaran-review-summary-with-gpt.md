# LLMを活用した宿泊レビュー要約 - モデル比較とプロンプトエンジニアリング

## はじめに

宿泊プラットフォームには膨大なレビューが蓄積されます。これらのレビューを一目で把握できるように要約すれば、ユーザー体験が大幅に改善されるでしょう。本記事では、OpenAIのGPTモデルを活用して日本語の宿泊レビューを要約し、**プロンプトエンジニアリングだけで安価なモデルの品質を引き上げた過程**を共有します。

核心的な流れは以下の通りです：

> シンプルなプロンプト → GPT-3.5（4点）、GPT-4（5点）
> → 詳細プロンプト適用 → GPT-3.5（2点に低下）
> → GPT-4出力をOne-shot例として活用 → GPT-3.5（5点達成）

## 1. じゃらんレビューのクローリング

要約するレビューデータを確保するため、日本の宿泊予約プラットフォームじゃらん（jalan.net）からレビューをクローリングしました。Seleniumでページをレンダリングし、BeautifulSoupでパースする方式です。

```python
from bs4 import BeautifulSoup
from selenium import webdriver

driver = webdriver.Chrome()
driver.get(url)

soup = BeautifulSoup(driver.page_source, 'html.parser')
containers = soup.select('.jlnpc-kuchikomiCassette')
```

各レビューカセット（`.jlnpc-kuchikomiCassette`）から以下の情報を抽出します：

- **レビュー本文**（`.jlnpc-kuchikomiCassette__postBody`）
- **総合評価**（`.jlnpc-kuchikomiCassette__totalRate`）
- **宿泊日**（「2025年12月」形式）
- **詳細評点** - 部屋、風呂、料理、接客・サービス、清潔感

詳細評点は`dt/dd`ペアで構成された評点リストから抽出し、この方法が失敗した場合はテキストから正規表現でフォールバック抽出します。

```python
def extract_ratings(container):
    ratings = {}
    rate_list = container.select_one('.jlnpc-kuchikomiCassette__rateList')
    if rate_list:
        dts = rate_list.find_all('dt')
        for dt in dts:
            label = dt.get_text(strip=True)
            dd = dt.find_next_sibling('dd')
            # RATING_MAPで日本語ラベル → 英語フィールド名にマッピング
            ...
    return ratings
```

ページネーションは「次へ」ボタンをクリックして最大5ページまで巡回します。3つの宿泊施設のレビューをそれぞれJSONファイルとして保存しました。

## 2. レビューデータの前処理

まず、JSON形式のレビューデータを前処理します。直近6ヶ月以内のレビューのみをフィルタリングし、星5つ（ポジティブ）とそれ以外（ネガティブ）に分類しました。各レビューは`[REVIEW_START]`と`[REVIEW_END]`タグで囲み、LLMがレビューの境界を明確に認識できるようにしました。

```python
for r in review_list:
    # 直近6ヶ月フィルタリング
    if review_date < date_boundary:
        continue

    if r['star'] == 5:
        reviews_good.append('[REVIEW_START]' + r['review'] + '[REVIEW_END]')
    else:
        reviews_bad.append('[REVIEW_START]' + r['review'] + '[REVIEW_END]')
```

## 3. Baselineプロンプトでモデル比較

最もシンプルなプロンプトから始めました。

```
以下の宿泊施設のレビューを5行くらいのボリュームで、まとめてください：
```

### GPT-3.5の結果

GPT-3.5は個別のレビューに番号を付けて列挙する方式で要約しました。各レビューの内容を縮約しましたが、**統合的な要約というよりはレビューの抜粋に近い結果**でした。

### GPT-4の結果

一方、GPT-4はテーマ別に情報を統合しました。食事、スパ、スタッフ対応、客室、混雑度などのカテゴリに分けて、全体のレビュー傾向を一つの要約にまとめました。

### G-Evalで品質評価

要約の品質を定量的に比較するため、[G-Eval](https://arxiv.org/abs/2303.16634)論文の評価方式を適用しました。Coherence（一貫性）指標を1〜5点のスケールで評価します。

```python
evaluation_prompt = f"""
Evaluation Criteria:
Coherence (1-5): The collective quality of all sentences.
A high-quality summary should be well-structured and organized,
and not just a heap of related sentences.
...
Source Text: {document}
Summary: {summary}
"""
```

| モデル | Coherenceスコア | 評価概要 |
|------|:-:|------|
| GPT-3.5 | 4 | 番号順に整理されているが、統合された一つの要約ではなく個別ミニ要約の羅列 |
| GPT-4 | 5 | テーマ別に構造化されており、一貫した流れを持つ |

GPT-4が確実に優れた結果を示しましたが、コスト差が大きいです。GPT-3.5の品質を引き上げる方法はないでしょうか？

## 4. 詳細プロンプト - むしろ逆効果

プロンプトに具体的な条件を追加してみました。

```
あなたは要約の専門家です。

要約結果は以下の条件を満たす必要があります：
1. すべての文は必ず丁寧語（です・ます体）で終えること
2. 宿泊施設を紹介するトーン＆マナーで作成すること
3. 要約結果は最低2文、最大5文で作成すること
```

良い例と悪い例まで提示してトーン＆マナーを明確に指定しました。しかし、結果は予想外でした。

**GPT-3.5のCoherenceスコア：2点**

GPT-3.5は指示を適切に守れませんでした。要約ではなく原文レビューをほぼそのままコピーし、文末に「ステキでしたです」「思いましたです」のような不自然な表現が現れました。条件が複雑になると、かえって基本的な要約能力まで低下してしまったのです。

これは重要な教訓を与えてくれます：**モデルの能力を超える複雑な指示は逆効果になり得る。**

## 5. One-shotプロンプト（1回目の試み） - 同一データの罠

ここで核心的なアイデアが登場します。GPT-4でまず良い要約例を生成し、それをGPT-3.5のプロンプトにone-shot例として入れるというものです。

```python
# ステップ1：GPT-4で例の要約を生成
summary_1shot = summarize(reviews, PROMPT_ONE_SHOT,
                          model='gpt-4-turbo-2024-04-09')

# ステップ2：生成した例をGPT-3.5プロンプトに含める
prompt_1shot = f"""あなたは要約の専門家です。
...
以下はレビューと要約の例です。
例のレビュー：
{reviews_1shot}
例の要約結果：
{summary_1shot}

以下の宿泊施設のレビューを要約してください："""

# ステップ3：GPT-3.5で実行
result = summarize(reviews, prompt_1shot,
                   temperature=1.0, model='gpt-3.5-turbo-0125')
```

### 結果：Coherence 5点、しかし問題がある

G-Eval Coherenceスコアは5点を達成しましたが、実際の出力を見ると問題が明らかになります：

> 和室湖畔側のお部屋からの景色が素晴らしく、清潔感あふれる空間で、夕食と朝食のバイキングが大満足でした。特に子供たちは屋上スパで大満足しており、貴重な家族の時間を過ごすことができました。スタッフの対応も温かく、良い旅行になりました。

この結果は**レビュー要約ではなく、実際にレビューを書いているかのよう**になっています。「〜大満足でした」「〜良い旅行になりました」は宿泊客本人の感想であり、「〜という評価です」「〜という声が多いです」のような要約トーンではありません。

原因は**one-shot例のレビューと実際の要約対象レビューが同一データ**だったためです。`reviews_1shot`と`reviews`がどちらも同じ`reviews.json`から取得していたため、モデルは例のトーンを学習するのではなく、既に見たレビュー内容をそのまま繰り返してしまったのです。

---

## 6. One-shotプロンプト（2回目の試み） - 別データで解決

問題を解決するため、2つの変更を行いました：

1. **one-shot例を別のレビュー/要約ペアでハードコーディング** - 要約対象とは完全に異なるデータを使用
2. **プロンプトに要約トーンと構造を明示** - 「〜という評価です」パターンと要約順序を指定

```python
# 要約対象とは異なる別の例データ
example_reviews = """[REVIEW_START]部屋は広くて快適でした。防音もしっかりしていて...[REVIEW_END]
[REVIEW_START]駅から近くてアクセスが便利でした。フロントの方も親切で...[REVIEW_END]
..."""

# 望むトーンの要約例を直接作成
example_summary = """部屋が広く清潔で、防音もしっかりしているという評価が多いです。
駅からのアクセスが便利で、フロントの対応も親切だという声が多く見られます。..."""

prompt_1shot = f"""あなたは要約の専門家です。
...
4. 個別のレビューをそのまま書くのではなく、複数のレビューの傾向をまとめて
   「〜という評価です」「〜という声が多いです」のように書いてください。
5. 要約は「施設・部屋 → 食事 → サービス・スタッフ → 総合評価」の順で
   構成してください。

以下はレビューと要約の例です。
例のレビュー：
{example_reviews}
例の要約結果：
{example_summary}

以下の宿泊施設のレビューを要約してください："""
```

### 結果：temperature=1.0でもCoherence 5点

`temperature=1.0`の高いランダム性でも安定して5点を達成しました。別データの例と明確な構造指示が効果を発揮したのです。

核心的な違いは出力のトーンにあります。1回目の試みで「大満足でした」（直接体験した感想）だったものが、2回目の試みでは「〜という評価が多いです」（レビュー傾向の要約）に変わりました。

最終要約結果：

> 総合的に素晴らしい宿泊施設との評価です。部屋は清潔で景色も良く、家族連れや友人同士で利用するには最適です。夕食、朝食ともに品数が豊富で、特に天空スパは大変満足されました。また、フロントやスタッフの対応も良く、快適な旅行となりました。再訪したいという声も多く見受けられます。

## まとめ

| 方式 | モデル | Coherence | 備考 |
|------|------|:-:|------|
| Baselineプロンプト | GPT-3.5 | 4 | 個別レビューの羅列 |
| Baselineプロンプト | GPT-4 | 5 | テーマ別統合要約 |
| 詳細条件プロンプト | GPT-3.5 | 2 | 指示過多による品質低下 |
| One-shot（同一データ例） | GPT-3.5 | 5 | スコアは高いがレビュートーンで出力 |
| One-shot（別データ例） | GPT-3.5 | 5 | 要約トーンで正しく出力 |

今回の実験から得られた示唆は以下の通りです：

1. **複雑な指示よりも良い例が勝る** - モデルの能力を超える条件を並べるより、望む結果の例を一つ見せる方がはるかに効果的である。
2. **one-shot例と実際の入力は必ず異なるデータを使うべきである** - 同一データを使用するとモデルがパターンを学習できず、内容をそのまま繰り返してしまう。例は「どんなトーンで要約するか」を教えるものであり、内容をコピーする合図ではない。
3. **定量評価だけでは不十分である** - Coherenceスコアが5点でも、実際の出力トーンが意図と異なる場合がある。G-Evalは構造的一貫性を評価するが、トーン＆マナーの適切性は別途確認が必要である。
4. **評価の自動化が重要である** - G-EvalのようなLLMベースの評価を導入すれば、プロンプト変更の効果を定量的に比較でき、勘に頼らない改善が可能になる。
